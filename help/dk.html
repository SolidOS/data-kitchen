data-kitchen

A single-page shell that hosts multiple apps under one menu, one auth pool, one theme, and one set of preferences.

Chrome (top row)

Menu: Home, Podz, Solidos, SolidResources, and DevTools — pick which app fills the display area. Apps marked keep-alive in the menu RDF stay mounted between switches, so things like a pod login or scroll position survive navigation back and forth.

Solidos is a two-pane workspace: a pod tree on the left, the SolidOS data browser on the right. Click the Solid logo (🌐-like icon) next to any item in the pod tree to load that resource in the right pane. See the Solidos help page for details.

SolidResources and DevTools are submenus — hover to see their items (website, FAQ, glossary, catalog; JSON-LD / SPARQL / SHACL playgrounds; an RDF validator). Each opens in an iframe inside the display area.

Search (🔍): inline search across DuckDuckGo, Google, Wikipedia, and others — your engine list is editable via Settings.

Calendar (🗓): the same data the dashboard's calendar shows, in a popout. Pre-fetches at page load so the events are ready when you click.

Help (?): you're looking at it. Today the help button shows this same page regardless of which app is active; per-app help pages live under help/<app>.html (e.g. help/podz.html, help/solidos.html).

Preferences (⚙): theme + font size. Per-app preferences (e.g. editor keybindings for podz) live inside the app itself. The gear also doubles as the logged-in indicator — see "Signing in" below.

Signing in

data-kitchen has no permanent login button. Login happens on demand: when a save (or any other authenticated fetch) hits a resource that requires sign-in, a login picker appears for the duration of the flow and disappears again once you're authenticated.

Once a session is active, the Preferences gear turns green; hovering it surfaces your WebID as the tooltip. Click the gear to reach Settings as usual — your session details and per-issuer logout will live there in the upcoming multi-issuer UI. (Today, all signed-in sessions share one default issuer configured under <sol-default default-issuer="…">.)

Behind the scenes: every widget that saves goes through solFetch, which dispatches sol-auth-needed on a 401. The chrome's hidden <sol-login> catches the event, opens an OIDC popup, and resolves the promise so the original save retries automatically.

Settings page

Lists every editable widget on the page. Expanding a panel lazy-mounts its editor; saves write back to the source TTL. The list starts with every panel collapsed.

Customizing the Data Kitchen

Copy installFolder/config.default.json to installFolder/config.json. Then edit the following fields:

Window size and position

Position can be either "center" or an integer (no quotes).
Start Page
Leave as an empty sting to start with the Data Kitchen intro page or set it to any URI.
LOCAL_BASE
This variable allows you to use "./" as a shortcut in the dataBrowser. If LOCAL_BASE is set to "file:///home/me/", a request in the dataBrowser URI bar for ./foo.txt will find the file "file:///home/me/foo.txt". You could have gotten to the same place by using that full URI. If this variable is missing or empty, it will default to the "myPod" folder of your installation folder.
REMOTE_BASE
This variable allows you to use "/" as a shortcut in the dataBrowser. If REMOTE_BASE is set to "https://me.example.org/public/", a request in the dataBrowser URI bar for "/foo.txt" will look for the resource at "https://me.example.org/public/foo.txt". You could have gotten to the same place by using that full URI. If this variable is missing or empty, you will not be able to use this shortcut.
Customize Bookmarks and Menus
Edit the bookmarks as you wish. URIs will open in the DataBrowser. Shortcuts like ./, /, and @ are okay.

Using URI shortcuts

In the dataBrowser URI bar, the start of the URI and the end of the URI determine how the URI will be interpreted. Shortcuts use the LOCAL_BASE and REMOTE_BASE variables, see "How to customize Data Kitchen" for details on how to set them.

URI starts with

  • http - absolute remote or localhost web address
  • file - absolute local file system address
  • app - absolute in-browser address (e.g. IndexedDB)
  • / - remote address relative to REMOTE_BASE
  • ./ - local file system address relative to LOCAL_BASE
  • @ - address of an ontology (see below)
URI ends with
  • / - URI is a container (folder,directory)
  • anything else - URI is a resource (file,document)

Ontology shortcuts

  • If the URI is in the form @prefix:term where prefix is an RDF prefix known to the Solid namespace, it will be expanded to the full URI. For example, @ldp:Container will bring you to http://www.w3.org/ns/ldp#Container. If you want to start out at the top level of the ontology, leave off the colon and term. For example, "@ldp" brings you to http://www.w3.org/ns/ldp

Some examples

  • if REMOTE_BASE is set to "https://you.example.org/public/" the URI "/foo" will expand to "https://you.example.org/public/foo"
  • if LOCAL_BASE is set to "file:///home/me/data/" the URI "./foo" will expand to "file:///home/me/data/foo"

Setting up a local profile

The dataBrowser expects to find key Solid resources (profile, settings, etc.) in the root folder of your file system. If it does not find them, the dataBrowser will behave differently. If you want your preferences to be honored (e.g. are you a Developer or PowerUser?), copy the following folders from the kitchen-data/myPod folder:

  • common (folder and contents)
  • profile (folder and contents)
  • settings (folder and contents)
  • .well-known (folder and contents)
  • .acl (file)

You can, if you wish use your profile and other resources from a remote pod instead of the supplied items. If you do this, make sure that addresses are relative. Also, make sure that the top level .acl gives access to .

You can also edit these files as you would on a pod.

You can also add other items (e.g. inbox) for special purposes.

Solid Login & Your Pod Library

Open Media Player can keep your library (genres, artists, playlists) on your own Solid pod instead of only in this browser. Logging in is optional — the player works fully without it using a local library.

The Log in button

Login lives inside the toolbar's ⋮ menu — open it and the top item is the Log in chip. Click it to pick a Solid identity provider (issuer) from the dropdown, or type your own issuer URL. You are redirected to that provider to authenticate, then returned here. When signed in the chip shows your WebID and becomes Log out; the ⋮ menu button itself turns green and adopts your WebID as its hover tooltip so session state is visible without opening the menu.

The OIDC sign-in is a real page redirect — the app reloads on the way out and back. That is normal; you briefly see the local library, then your pod library swaps in while staying signed in.

What happens when you sign in

  1. Resolve. Your WebID profile is read for a public type index, and that index is checked for an Open Media library registration (solid:forClass mo:Release). If found, that library loads.
  2. Remembered location. If the type index has no registration but you created a library here before, the player uses the location it remembered for your WebID.
  3. Create (bootstrap). Otherwise you are asked whether to create one (see below).

Creating a library on your pod (bootstrap)

  1. Choose where. You get a numbered list of every storage your profile advertises, plus the option to type any container URL. An open-media-player/ folder is created inside the one you pick.
  2. Confirm. You confirm the exact path before anything is written. Only data files are written — never app code.
  3. Type index (optional). If your profile has no public type index, you are asked whether to create one and link it to your profile so the library is auto-discovered next time. If your pod refuses that write, the library is still created and its location is remembered locally instead.
  4. Seeding. Your local agents.ttl and genres.ttl (your artists & genres) are copied into the new pod library so it isn’t empty. Releases and playlists start empty — add or convert them on the pod.

Every pod write during bootstrap is best-effort: if the type index or profile can’t be written, you’re told and that step is skipped — the library itself still works.

Viewing without logging in

If your pod library’s permissions allow public (unauthenticated) read, it is shown read-only when you are logged out — the player remembers its location and tries it at startup. The status bar says “Pod library loaded (read-only — log in to save changes)”. Logging in then upgrades the same library to read/write. If the pod is private, you transparently get the local library until you log in.

Logging out

Log out reverts to your local library; your previously enabled local libraries are restored. Nothing on the pod is changed.

Folder rename / migration

The pod folder is open-media-player/. An earlier version used open-media-browser/; on login the player performs a one-time, non-destructive copy of an old-named library into the new folder and repoints the type-index registration. The old folder is left in place.

Requirements & notes

TopicDetail
ServersBuilt and tested against Community Solid Server (CSS). NSS / ESS should work to the extent they support standard LDP PUT and application/sparql-update PATCH; anything refused degrades gracefully.
Auth libraryThe page must provide @inrupt/solid-client-authn-browser as a global. The bundled dev page (index.html) loads it; a host embedding <ia-player> supplies its own. Without it the button still shows; only an actual login errors.
Public readTo see the library logged out, grant the open-media-player/ container public Read in your pod’s access controls.
Local vs podThe pod library is never saved into this browser’s library list (so a logged-out reload can’t fail on it); only its URL is remembered to re-find it.
Known limitation. A dedicated “log in to save / keep changes in this browser only” prompt when a write is rejected is not yet implemented. For now, writes while logged out (or to a private pod) simply fail with a status message; log in to save.

Troubleshooting

  • Empty library after login — make sure you stayed signed in (the button shows Log out). If it shows Log in, the session didn’t restore; try logging in again.
  • It picked the wrong storage — re-run: log out, log in, and choose a different entry (or paste an exact container URL) at the storage prompt.
  • Writes fail — confirm you’re logged in and that your account has Write on the open-media-player/ container.
  • Can’t see it logged out — the container isn’t public-readable; add public Read in your pod’s access controls.
